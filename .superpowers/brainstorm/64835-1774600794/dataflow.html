数据流设计

一条消息从用户输入到最终响应的完整生命周期

1

📥 Channel 接收消息

CLI 或 Telegram Bot 接收到用户输入，统一封装为 Message{ChannelID, UserID, Text, Files, ReplyTo}，推送到 Orchestrator。

↓
2

🔍 Orchestrator：加载 Session + 注入记忆

根据 ChannelID+UserID 查找或创建 Session。从 Memory 层加载三类记忆注入 system prompt：项目记忆（当前工作区）+ 用户偏好 + 近期对话摘要。

↓
3

🎯 Orchestrator：路由到 Agent

分析消息意图，选择 Agent 类型：

🔨 CoderAgent

编写/修改/调试代码，拥有全部 13 个工具

👁 ReviewerAgent

代码审查，只有 read_file、git_diff、git_log 等只读工具

🔎 ResearcherAgent

技术调研，search、download、read 工具

↓
4

🔄 Agent ReAct 循环（现有逻辑保留）

LLM 推理 → 工具调用（并发执行）→ 观察结果 → 迭代，直到 finish_reason=stop。流式输出实时回传给 Channel。

↓
5

🧠 Memory 层：异步更新记忆

任务完成后，后台异步执行：① 更新项目记忆（检测到新文件/新技术栈）② 提取用户偏好变化 ③ 将本次对话摘要写入 SQLite（超过阈值时压缩历史）

↓
6

📤 Channel 响应

CLI：流式打印到终端（现有行为不变）。Telegram：分块发送文本，附件作为文件发送，长任务完成后推送通知消息。

关键设计：记忆注入时机

记忆在 步骤 2 注入 system prompt，Agent 天然感知上下文，无需改动 ReAct 循环。更新在 步骤 5 异步执行，不阻塞响应速度。

关键设计：流式响应与 Telegram

CLI 流式实时输出；Telegram 则积累内容、每隔 1-2 秒 edit_message 刷新，任务完成后发一条新消息触发通知。
