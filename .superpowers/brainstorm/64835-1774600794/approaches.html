三种架构方案

选择最适合 Mini-Code → OpenClaw 演进的整体架构方式

A

单体扩展（Monolithic Extension）

在现有 ClawEngine 和 main.go 上直接叠加新功能。Telegram 作为一个额外的输入输出适配器挂在旁边，记忆系统作为 Engine 的一个字段，多 Agent 通过工厂函数创建多个 Engine 实例。

优点

  • 改动最小，最快交付
  • 无需重构现有代码

缺点

  • engine.go 会变成"上帝类"（1000+ 行）
  • CLI 和 Telegram 逻辑耦合，难以独立测试
  • 多 Agent 调度混在 Engine 中，扩展困难

适合：快速验证，不打算长期维护

B

分层架构（Layered Architecture）⭐ 推荐

引入清晰的分层：Channel 层（CLI + Telegram 统一为消息接口）→ Orchestrator 层（路由、会话管理、Agent 调度）→ Agent 层（角色专化 Agent）→ Tool + Memory 层（共享基础设施）。

优点

  • 每层职责单一，可独立开发测试
  • 新增 Channel（如 WhatsApp）只需实现接口
  • Memory 和 Tool 被所有 Agent 共享，无重复
  • 与 OpenClaw 原始架构一致，学习成本低

缺点

  • 需要对现有代码做中等程度重构
  • 初期接口定义工作量较大

适合：长期维护，追求可扩展性

C

微内核（Microkernel / Plugin）

将现有 ClawEngine 作为"内核"保持不变，所有新功能（Telegram、Memory、多 Agent）以插件形式注册进来，通过事件总线（Event Bus）松耦合通信。

优点

  • 内核稳定，插件可热插拔
  • 理论上扩展性最强

缺点

  • 事件总线设计复杂，调试困难
  • Go 语言不像动态语言那样天然支持插件
  • 对于当前规模过度设计

适合：平台级产品，规模远超当前需求

我的推荐：B 方案 — 分层架构在改动量和可维护性之间取得最佳平衡。现有 ClawEngine 几乎不需要改动（只需实现 Agent 接口），新增代码都在新的包中，风险可控。
